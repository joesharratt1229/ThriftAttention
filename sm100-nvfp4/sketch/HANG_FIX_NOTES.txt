FA4 SM100 attention kernel hang: diagnosis and fix
===================================================
Date: 2026-07-03
File changed: fp4_attention_sm100.cu (correction_loop, 2 added mbarrier waits)

Symptom
-------
python3 run_fp4_attention.py hung intermittently inside
nvfp4_sm100_attention_kernel (no error, kernel never returns; Ctrl+C does
not kill the process cleanly). Reproduced 4/5 runs at the default shape
(q_len=256, kv_len=4096, single CTA), more often at larger kv_len.
Additionally, "successful" runs were nondeterministic: the same seed and
shape produced different output statistics run to run.

Root cause
----------
mbarrier parity bug in correction_loop's first StatsFull wait.

The correction warps' stats loop started with:

    int stats_phase = 1;
    for (int iter = 1; iter < kv_iters; iter++) {
        ...
        mbarrier_wait(... MbarId::StatsFull, stage), stats_phase);  // parity 1

The very first wait is on parity 1 of a freshly initialized mbarrier.
mbarrier parity waits complete whenever the barrier's phase bit differs
from the parity argument. A fresh barrier has phase bit 0, so a wait on
parity 1 falls through IMMEDIATELY -- a fresh barrier is indistinguishable
from "phase 1 already completed" under parity semantics. (issue_k/issue_v
exploit exactly this on KVEmpty deliberately, as a producer start state;
in correction_loop it was unintended.)

Consequence chain:
1. Correction warps never actually wait for softmax iteration-1 stats;
   they free-run one iteration ahead of softmax for the whole kernel.
2. Their PO arrivals (128 lanes per stage per iteration) therefore land
   one iteration early. The PO mbarrier counts anonymous arrivals
   (256 = 128 softmax + 128 correction per phase), so each PO phase can
   now complete WITHOUT the slowest softmax warp's 32 arrivals.
3. The MMA warp's PO waits pass early, so it commits SFull phases ahead
   of the softmax warps' intended schedule.
4. Any softmax warp that momentarily falls 2+ phases behind SFull is then
   permanently parity-aliased: it waits on parity p after the barrier has
   completed an even number of additional phases, so the phase bit equals
   p again and the wait never completes. Kernel deadlocks.
5. Even when no warp fell 2 behind (no hang), correction read
   stats_acc_scale racing softmax's writes and rescaled the O accumulator
   with wrong/stale scales -- the source of the nondeterministic outputs.

Evidence: on a hung run, cuda-gdb plus per-warp heartbeat instrumentation
showed softmax warps 1-7 fully finished and exited, softmax warp 0 stuck
in the SFull wait for iteration 62 (parity 0) after all 64 SFull phases
had completed (phase bit back at 0), the load warp finished all 64 TMA
iterations, and the MMA warp already past the last KVFull wait -- a state
impossible unless PO phases completed without warp 0's arrivals.

How the bug was found (debugging method)
----------------------------------------
1. Reproduce cheaply: ran run_fp4_attention.py in a loop under `timeout`.
   The default shape (q=256, kv=4096, a single CTA) hung 4/5 runs, proving
   the deadlock is intra-CTA and timing-dependent, not a multi-CTA or
   size-only effect.

2. Snapshot the hung kernel: ptrace is blocked in this container, so
   instead of attaching to a hung process, the script was launched UNDER
   cuda-gdb (batch mode), sent SIGINT after 60 s, then `info cuda warps`
   and per-warp `bt` were dumped. Because the kernel is built with
   -lineinfo, each warp's blocked PC resolved to a source line:
     - softmax warp 0: stuck in mbarrier_wait at the SFull wait (softmax
       score load), while softmax warps 1-7 had EXITED the kernel;
     - correction warps 8-11: waiting on OFull (epilogue);
     - MMA warp 12: waiting on a KVFull slot mid-loop;
     - load warp 14: exited (all TMA loads issued).
   That state is impossible under the intended protocol -- softmax warps
   cannot finish all SFull phases while the MMA warp is still mid-loop --
   so some parity wait had to be passing spuriously.

3. Pin down the exact iteration: built an instrumented copy of the kernel
   with a per-warp heartbeat -- each warp's leader stores an encoded
   (iteration, step) marker into a __device__ global array before/after
   every mbarrier wait and arrive. The host launcher polled
   cudaStreamQuery on the launch stream and, after a 20 s timeout, read
   the array back with cudaMemcpyAsync on a separate non-blocking stream
   (a plain sync memcpy would have blocked behind the hung kernel) and
   printed it. The dump on a hung run showed:
     - load warp done (64/64 iterations issued);
     - softmax warps 1-7 at iter 63 step 3 (finished, exited);
     - softmax warp 0 at iter 62 step 1 (blocked in the SFull wait);
     - MMA warp past the final KVFull wait, i.e. it had ALREADY committed
       all 64 SFull phases.
   64 completed phases put the SFull phase bit back at parity 0 -- the
   same parity warp 0 was waiting on for phase 62. A waiter that falls
   two phases behind an mbarrier can never wake: confirmed parity
   aliasing.

4. Work backwards to the seed: SFull commits are gated by PO phases, and
   PO phases should be impossible to complete without every softmax
   lane's arrival (256 anonymous arrivals per phase = 128 softmax + 128
   correction). Auditing every wait for how extra arrivals could appear
   led to correction_loop's first StatsFull wait: parity 1 on a fresh
   barrier, which falls through instantly. That is the only wait in the
   kernel whose first use violates the "never wait more than one phase
   early" rule that parity waits require.

5. Verify the theory: added the two seeding waits (below) to the
   instrumented copy first -- the hang disappeared (10/10 passes where
   before ~80% hung), output became bit-identical across runs, and it
   matched fp32 SDPA within nvfp4 error. Only then was the same change
   applied to fp4_attention_sm100.cu and re-verified on the production
   build.

Fix
---
Seed the parity sequence: consume StatsFull phase 0 (softmax iteration
0's arrival, whose scale is 1.0 and unused) before entering the loop.
Added in correction_loop, right after the initial PO arrives:

    // Consume StatsFull phase 0 (softmax iter 0) so the first wait below
    // (parity 1) cannot fall through on a fresh mbarrier: a fresh barrier
    // is indistinguishable from "phase 1 complete" under parity waits.
    mbarrier_wait(mbar_base + mbar_offset(MbarId::StatsFull, 0), 0);
    mbarrier_wait(mbar_base + mbar_offset(MbarId::StatsFull, 1), 0);

Softmax iteration 0 always arrives StatsFull regardless of correction
progress (it is gated only on SFull), so these waits cannot deadlock.
After seeding, the induction holds for every barrier in the kernel: each
waiter is at most one phase early and never two phases late, which is
exactly the regime in which parity waits are sound.

Verification
------------
- Before: 4/5 hangs at q=256/kv=4096. After: 10/10 passes.
- After: 3/3 passes each at q=1024/kv=8192, q=2048/kv=16384,
  q=512/kv=32768; 2/2 at large shapes on the production build.
- Output is now bit-identical across repeated runs of the same seed
  (was nondeterministic before).
- Sanity vs fp32 torch SDPA at q=512/kv=4096: cosine similarity 0.956,
  mean abs err 0.006, max abs err 0.037 -- consistent with nvfp4
  quantization error.

Known remaining issues (not causes of this hang, not yet fixed)
---------------------------------------------------------------
1. quantise_nvfp4.cu launches its kernels on the default stream while the
   pack/attention kernels use the current Torch stream. Identical in the
   plain script (current stream == default stream), but an ordering bug
   if the extension is ever called under a non-default CUDA stream
   (e.g. torch.cuda.Stream, CUDA graphs). The quantise launchers should
   take and use the same cudaStream_t.
2. Softmax and correction warps do tcgen05.wait::st but no
   tcgen05.fence::before_thread_sync before their mbarrier_arrive calls
   (PO / OEpi). Consumers do have the matching fence::after_thread_sync.
   Per the PTX memory model the "before" fence should be added ahead of
   each arrive that publishes tmem stores to another warp.
